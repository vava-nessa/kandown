# Configure agent system for kandown

## Context

Set up proper agent documentation so AI agents can use Kandown correctly when working on this project. This includes:
- Creating AGENTS.md at project root with quick architecture overview
- Creating AGENT_KANDOWN.md at project root with full task system documentation
- Updating the CLI to create AGENT_KANDOWN.md during `kandown init`
- Setting up .kanban/ directory for this project itself (kandown with kandown)

## What was done

### Created AGENT_KANDOWN.md at project root
- Comprehensive documentation of the Kanban task system
- File formats for board.md and tasks/t-xxx.md
- Task lifecycle, mutation rules, ID generation
- Priority system, owner types, tools field
- Reference guide for AI agents

### Created AGENTS.md at project root
- Quick architecture overview of the Kandown project
- References to AGENT_KANDOWN.md with instructions to read it
- Tech stack summary
- CLI commands reference
- Owner type system explanation

### Updated bin/kandown.js
- Modified `appendAgentReference()` to reference `AGENT_KANDOWN.md` instead of `.kanban/AGENT.md`
- Modified `createAgentsFileIfMissing()` with same update
- Added logic to copy `AGENT_KANDOWN.md` to project root during `kandown init`
- Added success message for AGENT_KANDOWN.md creation

### Created templates/AGENT_KANDOWN.md
- Template version of AGENT_KANDOWN.md for use during `kandown init`

### Set up .kanban/ for kandown itself
- Initialized .kanban/ directory with board.md, AGENT.md, README.md, kanban.html, and tasks/
- Updated board.md with project-appropriate tasks
- This enables developing kandown WITH kandown

## Notes

The key insight is that AGENT_KANDOWN.md lives at the project root (where `kandown init` is run), not inside `.kanban/`. This is because the agent documentation is for AI agents working on the project, not specifically about the kanban files themselves.
