## Context

Suite de t-025. Le serveur local expose maintenant une REST API complète. Cette tâche modifie `src/lib/filesystem.ts` pour détecter si le web app tourne en mode serveur (`http://localhost`) et utiliser `fetch()` au lieu de la File System Access API pour toutes les opérations fichier.

**Principe de base :** Deux modes coexistent dans le même fichier :
- **Mode `file://`** : comportement actuel inchangé (File System Access API, folder picker, etc.)
- **Mode `server`** : détecté via `window.__KANDOWN_ROOT__` injecté par le serveur, toutes les I/O via `fetch('/api/...')`

Le mode `file://` reste totalement fonctionnel — l'utilisateur qui ouvre `kandown.html` directement n'est pas impacté.

## Subtasks

- [ ] Créer `isServerMode(): boolean` — retourne `true` si `window.__KANDOWN_ROOT__` est défini et non vide
- [ ] Créer `getServerRoot(): string` — retourne `window.__KANDOWN_ROOT__` (chemin absolu de `.kandown/`) castée en string
- [ ] Implémenter `serverReadBoard(): Promise<string>` — `GET /api/board` → retourne le markdown brut
- [ ] Implémenter `serverWriteBoard(content: string): Promise<void>` — `PUT /api/board`
- [ ] Implémenter `serverReadConfig(): Promise<KandownConfig>` — `GET /api/config` → retourne le JSON parsé
- [ ] Implémenter `serverWriteConfig(config: KandownConfig): Promise<void>` — `PUT /api/config`
- [ ] Implémenter `serverListTasks(): Promise<string[]>` — `GET /api/tasks` → retourne liste d'IDs
- [ ] Implémenter `serverReadTask(id: string): Promise<string>` — `GET /api/tasks/:id`
- [ ] Implémenter `serverWriteTask(id: string, content: string): Promise<void>` — `PUT /api/tasks/:id`
- [ ] Implémenter `serverDeleteTask(id: string): Promise<void>` — `DELETE /api/tasks/:id`
- [ ] Modifier les fonctions publiques existantes pour brancher sur le mode serveur quand applicable (voir spec ci-dessous)
- [ ] Typer `window.__KANDOWN_ROOT__` dans un fichier `.d.ts` ou inline pour éviter les erreurs TypeScript

## Spec technique : Déclaration TypeScript de window.__KANDOWN_ROOT__

Ajouter dans `src/types/global.d.ts` (créer si inexistant) :
```typescript
interface Window {
  __KANDOWN_ROOT__?: string;
}
```

## Spec technique : isServerMode()

```typescript
export function isServerMode(): boolean {
  return typeof window !== 'undefined' && 
         typeof window.__KANDOWN_ROOT__ === 'string' && 
         window.__KANDOWN_ROOT__.length > 0;
}
```

## Spec technique : Fetch helpers

```typescript
const API_BASE = ''; // relatif — fonctionne sur n'importe quel port

async function apiFetch(path: string, options?: RequestInit): Promise<Response> {
  const res = await fetch(`${API_BASE}${path}`, options);
  if (!res.ok) throw new Error(`API ${options?.method ?? 'GET'} ${path} → ${res.status}`);
  return res;
}
```

## Spec technique : Quelles fonctions modifier

Uniquement les fonctions utilisées par `store.ts` pour les I/O de fichiers. Voici la liste exacte et comment les brancher :

### `readBoardFile(handle)`
```typescript
export async function readBoardFile(handle: ...) {
  if (isServerMode()) return serverReadBoard();
  // ... code existant inchangé
}
```

### `writeBoardFile(handle, content)`
```typescript
export async function writeBoardFile(handle: ..., content: string) {
  if (isServerMode()) return serverWriteBoard(content);
  // ... code existant
}
```

### `readConfigFile(handle)` / `writeConfigFile(handle, config)`
Même pattern : si `isServerMode()`, utiliser `serverReadConfig()`/`serverWriteConfig()`.

### `readTaskFile(tasksHandle, id)` / `writeTaskFile(...)` / `deleteTaskFile(...)`
Même pattern.

### `pickProjectDirectory()` / `getKandownHandle()`
**Ne pas modifier.** Ces fonctions ne sont appelées qu'en mode `file://`. En mode serveur, `store.ts` bypasse complètement le folder picker (voir t-027).

### `supportsFileSystemAccess()`
Ajouter un cas : en mode serveur, toujours retourner `true` (le "support" vient du serveur).
```typescript
export function supportsFileSystemAccess(): boolean {
  if (isServerMode()) return true;
  return 'showDirectoryPicker' in window;
}
```

## Spec technique : Gestion d'erreurs fetch

Les fonctions serveur doivent propager les erreurs clairement :
- 404 → lancer `Error('Not found: ' + path)` (le store le catchera et affichera un toast)
- 500 → lancer `Error('Server error: ' + responseText)`
- Network error → lancer l'erreur fetch native

## Notes importantes pour l'agent

- **Ne jamais casser le mode `file://`** — toutes les modifications sont additive avec `if (isServerMode())`
- Lire `src/lib/store.ts` pour comprendre exactement quelles fonctions de `filesystem.ts` sont appelées et avec quels arguments
- Lire `src/lib/types.ts` pour le type `KandownConfig` exact
- Les `handle` (FileSystemDirectoryHandle) ne sont jamais undefined en mode serveur — les fonctions branchent avant de les utiliser donc les handles peuvent être ignorés (passer `null as any` si nécessaire)
- Ne pas modifier `saveRecentProject` / `listRecentProjects` — le mode serveur n'utilise pas l'IndexedDB pour les projets récents
- Ajouter des JSDoc à toutes les nouvelles fonctions
