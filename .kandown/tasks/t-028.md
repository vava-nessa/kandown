## Spec technique : Live-reload via Server-Sent Events (optionnel mais recommandé)

Quand le TUI board modifie `board.md` (ou quand un AI agent le modifie), le browser devrait voir les changements sans F5.

**Côté serveur** (dans `bin/kandown.js`) :

```js
// Endpoint SSE
// GET /api/watch → stream d'événements
function handleWatch(req, res, kandownDir) {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
    ...corsHeaders
  });
  res.write('data: connected\n\n');
  
  const watcher = fs.watch(kandownDir, { recursive: true }, (event, filename) => {
    if (filename?.endsWith('.md') || filename?.endsWith('.json')) {
      res.write(`data: ${JSON.stringify({ event, file: filename })}\n\n`);
    }
  });
  
  req.on('close', () => watcher.close());
}
```

**Côté client** (dans `src/App.tsx` ou `src/lib/store.ts`) :

```typescript
// Si en mode serveur, écouter les changements de fichiers
if (isServerMode()) {
  const es = new EventSource('/api/watch');
  es.onmessage = (e) => {
    if (e.data !== 'connected') {
      store.getState().reloadBoard(); // action existante dans store.ts
    }
  };
  // Cleanup si le composant démonte (rare mais propre)
  return () => es.close();
}
```

**Important :** Ce live-reload est une amélioration bonus. Si l'implémentation complique trop les choses, la skipper et documenter comme "TODO" dans un commentaire.

## Spec technique : Gestion du watcher de fichiers dans store.ts

Le store a déjà `src/lib/watcher.ts` qui surveille les fichiers en mode `file://`. En mode serveur, le watcher natif du navigateur (FileSystemObserver) n'est pas disponible. Le SSE du serveur le remplace.

S'assurer que `watcher.ts` n'est **pas** initialisé en mode serveur (il utilise des APIs browser qui n'existent pas dans ce contexte ou qui nécessitent un handle).

## Spec technique : Arrêt propre (graceful shutdown)

```js
process.on('SIGINT', () => {
  server.close(() => {
    log('\n→ Server stopped.');
    process.exit(0);
  });
});
```

Mais attention : le TUI board (Ink) gère aussi les signaux. S'assurer que le shutdown est coordonné — le TUI doit quitter proprement (restaurer le terminal) avant que le serveur ferme.

## Cas limites à vérifier

| Situation | Comportement attendu |
|-----------|---------------------|
| Port 2048 déjà occupé | Message clair "Port 2048 occupé, essai sur 2049…", utilise le prochain port libre |
| Serveur arrêté, browser toujours ouvert | La page affiche une erreur réseau via toast, propose de relancer `kandown` |
| `kandown.html` absent dans `.kandown/` | Message d'erreur "Relancez `kandown init`" |
| Pas de `.kandown/` dans le dossier courant | Message d'erreur clair avec suggestion `kandown init` |
| Deux instances `kandown` en parallèle | Chacune prend un port différent — OK, pas de conflit |
| Board modifié par un AI agent pendant que le browser est ouvert | Live-reload SSE rafraîchit le board automatiquement |
