# Fix kandown CLI startup parser crash

## Context

Running `kandown` prints a large minified parser-error object instead of launching cleanly. Reproduce the CLI failure, verify the browser-served page, patch the root causes, then release the fixed build.

## Subtasks

- [x] Reproduce the command failure locally
  report: `node bin/kandown.js --port 2055` starts the TUI, but the served browser page renders raw bundled parser text and logs `SyntaxError: Unexpected token '.'`.
- [x] Inspect the served web UI with browser tooling
  report: Chrome DevTools MCP was profile-locked, so the in-app browser inspected `http://localhost:2055`; the parsed script showed injected `window.__KANDOWN_ROOT__` text inside the bundle near a parser error string containing literal `</head>`.
- [x] Patch the CLI or web bundle root cause
  report: Added `injectServerRoot()` in `bin/kandown.js` so server-mode injection targets the last real `</head>` and escapes `<` in the JSON-encoded project path.
- [x] Patch the single-file regex repair
  report: Added a Vite post-build repair plugin that restores `(?<` regex lookbehind openers after esbuild escapes them as invalid `(?\x3C` sequences in the inlined Shiki grammar bundle.
- [x] Build and verify the command/runtime flow
  report: `pnpm build` passed. A fresh temp project launched with `node bin/kandown.js --port 2057`, rendered the web board without raw parser text, and had no browser console errors. The generated standalone `kandown.html` also opened via `file://` without console errors.
- [x] Bump minor version, update changelog, commit, tag, push, and verify publish
  report: Corrected the release to v0.4.0 because the unreleased commits include features, updated `CHANGELOG.md`, rebuilt with `pnpm build`, and prepared the release commit/tag workflow.

## Notes

Started from a clean `main` worktree. The exact user-facing browser symptom was reproduced through the in-app browser after Chrome DevTools MCP reported a profile lock.
