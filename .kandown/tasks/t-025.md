## Context

Suite de t-024. Le serveur HTTP est créé et route `/api/*` vers `handleApi()`. Cette tâche implémente tous les endpoints REST qui permettent au web app de lire et écrire les fichiers markdown du projet sans la File System Access API.

Tous les endpoints opèrent sur le `kandownDir` passé au serveur (chemin absolu de `.kandown/` du projet courant). Les lectures/écritures utilisent `fs.readFileSync`/`fs.writeFileSync` — synchrone pour simplifier, le serveur est monothread et local.

**Contrainte forte :** Zéro nouvelle dépendance. Uniquement `fs`, `path`, `http` natifs.

## Subtasks

- [ ] Implémenter `GET /api/config` → lire `.kandown/kandown.json`, retourner JSON
- [ ] Implémenter `PUT /api/config` → écrire body JSON dans `.kandown/kandown.json`
- [ ] Implémenter `GET /api/board` → lire `.kandown/board.md`, retourner `text/plain`
- [ ] Implémenter `PUT /api/board` → écrire body string dans `.kandown/board.md`
- [ ] Implémenter `GET /api/tasks` → lister les fichiers dans `.kandown/tasks/`, retourner tableau JSON des IDs
- [ ] Implémenter `GET /api/tasks/:id` → lire `.kandown/tasks/:id.md`, retourner `text/plain`
- [ ] Implémenter `PUT /api/tasks/:id` → écrire body string dans `.kandown/tasks/:id.md` (créer si inexistant)
- [ ] Implémenter `DELETE /api/tasks/:id` → supprimer `.kandown/tasks/:id.md`
- [ ] Gérer les erreurs : fichier inexistant → 404, erreur lecture/écriture → 500 avec message
- [ ] Lire le body des requêtes PUT via les events `data`/`end` du stream (pas de body-parser)

## Spec technique : Tous les endpoints

### `GET /api/config`
```
Response: 200 application/json
Body: contenu de kandown.json parsé en JSON
Erreur: 404 si kandown.json n'existe pas
```

### `PUT /api/config`
```
Request body: JSON string
Response: 200 {"ok": true}
Action: écrire le body tel quel dans kandown.json (valider que c'est du JSON valide, sinon 400)
```

### `GET /api/board`
```
Response: 200 text/plain; charset=utf-8
Body: contenu brut de board.md
Erreur: 404 si board.md n'existe pas
```

### `PUT /api/board`
```
Request body: contenu markdown brut (text/plain)
Response: 200 {"ok": true}
Action: écrire le body dans board.md
```

### `GET /api/tasks`
```
Response: 200 application/json
Body: ["t-001", "t-002", ...] — IDs des fichiers dans tasks/ (sans extension .md)
Erreur: 200 [] si le dossier tasks/ n'existe pas encore
```

### `GET /api/tasks/:id`
```
Response: 200 text/plain; charset=utf-8
Body: contenu brut de tasks/:id.md
Erreur: 404 si le fichier n'existe pas
```

### `PUT /api/tasks/:id`
```
Request body: contenu markdown brut
Response: 200 {"ok": true}
Action: écrire dans tasks/:id.md, créer le dossier tasks/ si inexistant
```

### `DELETE /api/tasks/:id`
```
Response: 200 {"ok": true}
Erreur: 404 si le fichier n'existe pas
```

## Spec technique : Lecture du body (sans dépendance)

```js
function readBody(req) {
  return new Promise((resolve, reject) => {
    const chunks = [];
    req.on('data', chunk => chunks.push(chunk));
    req.on('end', () => resolve(Buffer.concat(chunks).toString('utf8')));
    req.on('error', reject);
  });
}
```

## Spec technique : Sécurité basique (path traversal)

Avant toute opération sur un `:id`, valider :
```js
// Refuser tout id qui contient '/', '..', ou des caractères hors [a-zA-Z0-9_-]
if (!/^[a-zA-Z0-9_-]+$/.test(id)) { res.writeHead(400); res.end('Invalid task id'); return; }
```

## Spec technique : Parsing de la route dans handleApi

```js
function handleApi(req, res, url, kandownDir) {
  const parts = url.pathname.replace('/api/', '').split('/');
  const resource = parts[0]; // 'board', 'config', 'tasks'
  const id = parts[1];       // pour /api/tasks/:id
  
  // Router par resource + method
  if (resource === 'board' && req.method === 'GET') return getBoard(res, kandownDir);
  if (resource === 'board' && req.method === 'PUT') return putBoard(req, res, kandownDir);
  // ... etc
}
```
