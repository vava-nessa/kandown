Créer les modules Node.js de base pour le CLI task launcher : lecture du board, registre d'agents AI, et lanceur de processus (tmux + direct).

## Context

Le CLI kandown a déjà un TUI (Ink 7) avec un écran settings. On ajoute un board interactif qui permet de lancer des agents AI sur des tasks. Ces modules sont les briques de base.

Le parser existant (`src/lib/parser.ts`) fonctionne sur des strings pures (pas de dépendance browser) → réutilisable côté Node directement.

## Subtasks

- [x] Créer `src/cli/lib/board-reader.ts` — wrapper Node fs autour de parseBoard/parseTaskFile + fonction moveTaskToColumn()
  report: Créé. Fonctions: readBoard, readTask, readAgentDoc (avec fallback COMPACT → full → AGENT.md), moveTaskToColumn (string manipulation pure, pas de re-sérialisation).
- [x] Créer `src/cli/lib/agents.ts` — registre d'agents (claude, codex, gemini, goose, aider, opencode), détection via `which`, construction de commandes
  report: Créé. 6 agents définis avec buildCommand, détection cachée via which, fonction buildPrompt qui construit systemPrompt + taskPrompt séparément.
- [x] Créer `src/cli/lib/launcher.ts` — détection tmux, split-pane ou exec direct, gestion du cycle de vie processus
  report: Créé. isInTmux() via $TMUX, tmux split-window -h -p 50, exec direct avec spawn + stdio inherit. Auto-move vers In Progress, context file dans /tmp, env vars KANDOWN_* exposés.

## Notes

- `moveTaskToColumn` doit manipuler board.md en string (regex pour trouver la ligne task, la déplacer sous le bon header de colonne)
- Détection agents : `execFileSync('which', [bin])` en try/catch, résultat caché
- Pour Claude Code : utiliser `--system-prompt` pour le doc compact + premier message comme arg
- tmux : `tmux split-window -h -p 50 'command'` si `process.env.TMUX` existe
- Sans tmux : quitter le TUI proprement (restore terminal), exec agent en foreground
