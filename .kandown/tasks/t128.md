# Replace Wysimark with BlockNote \(markdown\-pure adapter layer\)

## Context

Today the task drawer body is edited via `@wysimark/react` in `src/components/ui/MarkdownEditor.tsx`\. Wysimark is functional but dated and limited compared to modern block editors\. We want to migrate to **BlockNote** \([https\:\/\/github\.com\/TypeCellOS\/BlockNote](https://github.com/TypeCellOS/BlockNote)\) for a Notion\-like editing experience\: slash menu\, drag handles\, structured blocks\.

**Hard constraint** \: the `.kandown/tasks/*.md` files MUST remain pure\, idempotent\, git\-friendly markdown\. No HTML inline\, no JSON sidecar\, no metadata pollution\. The editor is a UI layer over markdown — markdown stays the source of truth\.

**Scope** \: ONLY the **task description body** \(the part rendered by `MarkdownEditor` inside the task drawer\)\. Title\, frontmatter \(priority\/tags\/assignee\/etc\)\, and subtasks checklist are NOT migrated and stay handled by their existing dedicated components\.

## Why BlockNote

- Built on Tiptap\/Prosemirror — robust\, battle\-tested

- Native helpers\: `blocksToMarkdownLossy()` and `tryParseMarkdownToBlocks()`

- Slash menu\, drag\-and\-drop\, block\-level cursor — modern UX

- Configurable schema → we can restrict allowed blocks

## Strategy\: strict whitelist\, no pollution

We adopt the **whitelist stricte** approach decided with the user \:

- Only blocks with a 1\:1 markdown equivalent are exposed in the editor

- All other BlockNote features \(callouts\, columns\, file blocks\, text alignment\, text colors\, custom blocks\) are removed from the schema and slash menu

- Bundle size is NOT a concern \(user accepted the \~1MB hit\)

### Whitelisted blocks \(markdown\-native only\)

|BlockNote block|Markdown equivalent|
|---|---|
|Paragraph|Plain text|
|Heading 1\/2\/3|`#`\, `##`\, `###`|
|Bullet list item|`- `|
|Numbered list item|`1. `|
|Checklist item|`- [ ]` \/ `- [x]`|
|Quote|`> `|
|Code block|```` ```lang ````|
|Inline code|`` ` ``|
|Bold \/ Italic \/ Strike|`**` \/ `*` \/ `~~`|
|Link|`[text](url)`|
|Image|`![alt](url)`|
|Horizontal rule|`---`|

### Blocks explicitly disabled

- Callout \/ alert blocks

- Column \/ multi\-column layouts

- Text alignment \(left\/center\/right\/justify\)

- Text colors and background colors

- File \/ video \/ audio embeds

- Tables \(BlockNote tables don\'t round\-trip cleanly to GFM tables — TBD if added later as a phase 2\)

- All custom blocks

## The 4 anti\-pollution guardrails

These are the non\-negotiable safeguards to make the round\-trip clean \:

### 1\. Save only on dirty

Track `isDirty` state in the editor\. If the user opens a drawer and closes it without typing\, the file MUST NOT be touched\. No silent normalization on load\.

```ts
const [isDirty, setIsDirty] = useState(false);
editor.onChange(() => setIsDirty(true));
// in save handler:
if (!isDirty) return; // skip write entirely
```

### 2\. Frontmatter and H1 title parsed OUTSIDE BlockNote

The BlockNote editor receives ONLY the body content \(everything after frontmatter \+ the `# Title` H1\)\. Frontmatter is parsed\/serialized by the existing YAML logic\. The H1 title is handled by the title input\. BlockNote never sees them\, never touches them\.

This is critical \: if BlockNote were to round\-trip the frontmatter\, it would destroy YAML formatting\.

### 3\. Idempotence test in the build \/ CI

Add a script `scripts/check-md-idempotence.js` that \:

1. Reads every `.kandown/tasks/*.md` \(and `templates/example-tasks/*.md`\)

2. Splits frontmatter \+ H1 from body

3. Runs body through `tryParseMarkdownToBlocks()` then `blocksToMarkdownLossy()`

4. Compares byte\-for\-byte to original body

5. Fails the build if any file diverges

This catches regressions where a BlockNote update changes serialization\.

### 4\. One\-shot migration of existing tasks

Before merging\, run a migration script that pre\-normalizes all existing `.md` files through the BlockNote serializer pipeline once\. Commit the result as a single migration commit\. From there\, any future save is a no\-op for unchanged files\.

Without this step\, every existing task would diff on first open\.

## Non\-goals

- Tables \(defer to phase 2 task — GFM tables in BlockNote need investigation\)

- Mentions \/ autocomplete

- Real\-time collaboration features \(Yjs \/ Liveblocks integration BlockNote supports — not needed\, kandown is local\-first\)

- Custom block types

- Replacing the title input or subtasks checklist

## Risks \& open questions

- **Single\-file bundle** \: verify `kandown.html` still builds cleanly with BlockNote\'s CSS injection\. Mantine UI may cause issues with `vite-plugin-singlefile` — test early

- **CSS conflicts** \: BlockNote ships its own styles\. Need to scope or override carefully so it doesn\'t bleed into the rest of the app

- **GFM checklist support** \: confirm BlockNote\'s checklist block round\-trips to `- [ ]` \/ `- [x]` \(GFM extension\)\, not to `<input type="checkbox">`\. If not\, write a custom serializer override

- **Image handling** \: BlockNote supports image upload — for kandown \(local\-first\, no backend\) we restrict to URL\-only images \(paste markdown image syntax\)\. Verify this is configurable in the schema
