## Context

Suite de t-026. `filesystem.ts` sait maintenant lire/écrire via le serveur local. Cette tâche modifie `store.ts` et `App.tsx` pour que le web app s'initialise **automatiquement** sur le projet courant quand il tourne en mode serveur — sans folder picker, sans clic utilisateur.

Le flux actuel en mode `file://` :
1. App monte → affiche `EmptyState` (folder picker)
2. L'utilisateur clique "Ouvrir un projet" → `showDirectoryPicker()` → `openFolder()`
3. Le board charge

Le flux cible en mode serveur :
1. App monte → détecte `window.__KANDOWN_ROOT__` → appelle `openServerProject()` directement
2. Le board charge automatiquement — **zéro interaction utilisateur**

## Subtasks

- [ ] Créer `openServerProject()` dans `store.ts` — charge config + board sans handle ni folder picker
- [ ] Modifier `App.tsx` : au montage (`useEffect`), si `isServerMode()` → appeler `openServerProject()`
- [ ] Modifier `EmptyState.tsx` : ne pas afficher le folder picker en mode serveur (afficher un spinner "Chargement…" à la place)
- [ ] Gérer l'état d'erreur si le serveur n'est pas disponible : toast + message "Relancer `kandown`"
- [ ] Gérer le rechargement de page : si `isServerMode()` et le projet est déjà "ouvert" dans le store, ne pas re-déclencher `openServerProject()`

## Spec technique : openServerProject() dans store.ts

Cette fonction est l'équivalent de `openFolder()` mais pour le mode serveur. Elle doit :

1. Mettre `state.loading = true`
2. Appeler `readConfigFile(null)` — en mode serveur, `filesystem.ts` ignore le handle
3. Appeler `applyProjectTheme(config)` avec la config lue
4. Appeler `readBoardFile(null)` → parser le markdown → charger les colonnes et tasks
5. Mettre `state.projectName` à partir du basename du `KANDOWN_ROOT` (ex: `/projects/myapp/.kandown` → `myapp`)
6. Mettre `state.isOpen = true`, `state.loading = false`
7. En cas d'erreur : `state.loading = false`, appeler `toast('Impossible de charger le projet')`

```typescript
openServerProject: async () => {
  set({ loading: true });
  try {
    const config = await readConfigFile(null as any); // filesystem gère le mode serveur
    applyProjectTheme(config);
    const boardMd = await readBoardFile(null as any);
    const parsed = parseBoard(boardMd);
    // ... même logique que openFolder pour charger les colonnes
    set({ isOpen: true, loading: false, config, /* columns, etc. */ });
  } catch (err) {
    set({ loading: false });
    get().toast('Impossible de charger le projet. Relancez `kandown`.');
  }
},
```

**Important :** Regarder exactement comment `openFolder()` charge les données (quels champs il set dans le store) et reproduire le même comportement. Ne pas dériver — copier le pattern exact.

## Spec technique : Modification de App.tsx

Chercher le `useEffect` de montage dans `App.tsx` (ou en créer un si inexistant) :

```typescript
useEffect(() => {
  if (isServerMode()) {
    store.openServerProject();
  }
}, []); // eslint-disable-line react-hooks/exhaustive-deps
```

Ce useEffect doit être le premier qui s'exécute — avant tout autre effet qui dépendrait d'un projet chargé.

## Spec technique : Modification de EmptyState.tsx

Ajouter en haut du composant :

```typescript
if (isServerMode()) {
  return (
    <div className="..."> {/* même style que le loading state actuel */}
      <Spinner /> {/* ou un div animé existant */}
      <p>Chargement du projet…</p>
    </div>
  );
}
```

Utiliser les classes Tailwind / composants déjà présents dans EmptyState pour que ça soit cohérent visuellement. Ne pas créer un nouveau design.

## Spec technique : Rechargement / hot-reload

Si l'utilisateur recharge la page (`F5`) en mode serveur :
- `window.__KANDOWN_ROOT__` est re-injecté par le serveur à chaque requête `GET /`
- Le `useEffect` de montage se re-déclenche → `openServerProject()` se rappelle → OK

Pas de cas spécial à gérer.

## Spec technique : Mode file:// inchangé

Vérifier que le `useEffect` ne s'exécute **pas** quand `isServerMode()` est false. Le comportement existant (EmptyState avec folder picker) doit rester 100% fonctionnel.
