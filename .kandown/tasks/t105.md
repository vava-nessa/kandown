# Context

When the disk is full or the browser's storage quota is exceeded, file write operations in Kandown fail silently. The current implementation:

1. **Optimistically mutates state** (Zustand store) BEFORE the actual file write completes
2. **Catches the write error** and shows a toast, but does NOT restore all previous state
3. **Leaves the UI in an inconsistent state** — the user sees their change applied, but it's never persisted to disk

This means on disk-full scenarios, user data can appear to save but actually be lost on page reload.

**Affected operations:**
- `writeConfigFile` at `src/lib/filesystem.ts:126-131` — no error handling for `createWritable` or `write`
- `writeTaskFile` at `src/lib/filesystem.ts:168-178` — same issue
- `store.ts:350-360` (`updateConfig`) — optimistic state change then write, rollback only shows toast
- `store.ts:412-421` (`moveTask`) — optimistic column change then write, rollback exists but incomplete
- `store.ts:542-587` (`createTask`) — optimistic board change then write, rollback missing taskContents restoration

**Current behavior on disk full:**
1. User moves a task (drag-drop)
2. Zustand store updates optimistically (column changes immediately in UI)
3. `persistColumnOrder` calls `writeTaskFile` for each affected task
4. If any write fails (disk full, quota exceeded), `Promise.all` rejects
5. `moveTask` catches error, shows toast, calls `set({ columns })` with pre-mutation `columns`
6. UI snaps back — **BUT** `taskContents` cache and `searchMatches` are NOT rolled back
7. If page is reloaded before next successful write, the task is lost from the board

## Why This Needs Fixing

**Severity: Critical — Data Loss**

The user can make changes that appear to succeed in the UI but are never persisted to disk. On page reload, those changes are gone with no recovery path. Every write operation in the app needs proper error handling with complete rollback capability covering ALL Zustand state (columns, taskContents, searchMatches, config, etc).

## How to Fix

### Step 1: Create Error Types (`src/lib/errors.ts`)

```typescript
export class DiskFullError extends KandownError {
  constructor(path: string) {
    super(`Disk full or quota exceeded: cannot write to ${path}`, 'DISK_FULL');
    this.name = 'DiskFullError';
  }
}
```

### Step 2: Update Filesystem Write Functions

In `src/lib/filesystem.ts`, wrap all write operations:

```typescript
export async function writeConfigFile(kandownHandle, config) {
  try {
    const h = await kandownHandle.getFileHandle('kandown.json', { create: true });
    const w = await h.createWritable();
    await w.write(JSON.stringify(config, null, 2) + '\n');
    await w.close();
  } catch (e) {
    if (e.name === 'QuotaExceededError' || e.message?.includes('quota')) {
      throw new DiskFullError('kandown.json');
    }
    throw e;
  }
}
```

Same pattern for `writeTaskFile`. Also handle `ensureTasksDir` — if `getDirectoryHandle({ create: true })` throws due to quota, surface a clear error.

### Step 3: Create Retry Utility (`src/lib/retry.ts`)

```typescript
export async function withRetry<T>(
  operation: () => Promise<T>,
  options: {
    maxAttempts?: number;
    baseDelayMs?: number;
    retryableCheck?: (error: unknown) => boolean;
  } = {}
): Promise<T> {
  const { maxAttempts = 3, baseDelayMs = 1000, retryableCheck = isRetryableError } = options;
  let lastError: Error;

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return await operation();
    } catch (e) {
      lastError = e;
      if (attempt < maxAttempts && retryableCheck(e)) {
        await new Promise(r => setTimeout(r, baseDelayMs * Math.pow(2, attempt - 1)));
        continue;
      }
    }
  }
  throw lastError!;
}

function isRetryableError(error: unknown): boolean {
  if (error instanceof DiskFullError) return true;  // User might free space between retries
  if (error instanceof DOMException) {
    return error.name === 'NetworkError' || error.name === 'TimeoutError';
  }
  return false;
}
```

### Step 4: Fix Store Actions with Complete Rollback

Every mutating action follows this pattern:

```typescript
action: async (...) => {
  const { columns, taskContents, searchMatches, ... } = get();
  
  // 1. Capture ALL previous state
  const prev = { columns, taskContents, searchMatches };
  
  // 2. Optimistic update
  set({ columns: newColumns, taskContents: newContents });
  
  // 3. Write with retry
  try {
    await withRetry(() => persistColumnOrder(...));
  } catch (e) {
    // 4. FULL rollback
    set(prev);
    
    // 5. User-friendly error
    if (e instanceof DiskFullError) {
      get().toast('Disk is full — changes not saved. Free up space and try again.', 'error', 10000);
    } else {
      get().toast(`Failed to save: ${e.message}`, 'error');
    }
  }
}
```
