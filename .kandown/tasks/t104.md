# Context

Multiple store actions perform optimistic updates (mutating Zustand state before file writes confirm) but their rollback mechanisms are incomplete. When a file write fails:

1. **Columns state is rolled back** ✓
2. **taskContents cache is NOT rolled back** ✗
3. **searchMatches cache is NOT rolled back** ✗

This leaves the store in a state where the board appears correct (columns rolled back) but the caches contain stale or inconsistent data. Reloading the page would show different state than what the current UI shows.

**Affected actions:**

### `moveTask` (store.ts:394-423)
```typescript
moveTask: async (taskId, fromCol, toCol, toIndex) => {
  // ...
  set({ columns: newColumns });  // Optimistic
  // taskContents not updated optimistically (good)
  try {
    await persistColumnOrder(tasksDirHandle, affected, config.board.columns);
  } catch (e) {
    get().toast('Failed to save: ' + (e as Error).message, 'error');
    set({ columns });  // Rollback columns only
    // taskContents was never touched, so no issue here
  }
}
```
**Problem**: `persistColumnOrder` at line 192-215 uses `Promise.all` — if some writes succeed and others fail, board state is inconsistent with files.

### `createTask` (store.ts:542-587)
```typescript
// ...
const newContents = new Map(taskContents);
newContents.set(id, { frontmatter: fm, body });  // Line 578 - Cache updated BEFORE write
set({ taskContents: newContents });  // Line 579
// ...
try {
  await fsWriteTaskFile(tasksDirHandle, id, fm, body);  // Line 576
} catch (e) {
  get().toast('Failed to create: ' + (e as Error).message, 'error');
  set({ columns });  // Rollback columns
  // BUT taskContents is NOT rolled back — new task entry remains in cache
}
```

### `deleteTask` (store.ts:589-609)
```typescript
const newColumns = columns.map(c => ({ ...c, tasks: c.tasks.filter(t => t.id !== taskId) }));
set({ columns: newColumns });  // Line 592 - Optimistic
// ...
const newContents = new Map(taskContents);
newContents.delete(taskId);  // Line 597 - Cache mutated
const newMatches = new Map(searchMatches);
newMatches.delete(taskId);  // Line 598 - Cache mutated
set({ taskContents: newContents, searchMatches: newMatches });  // Line 600

try {
  await fsDeleteTaskFile(tasksDirHandle, taskId);  // Line 603
} catch (e) {
  get().toast('Failed to delete: ' + (e as Error).message, 'error');
  set({ columns });  // Rollback columns only
  // taskContents and searchMatches NOT rolled back — task appears deleted
  return;
}
```

### `reorderInColumn` (store.ts:425-439)
Same issue — no rollback for taskContents if writes fail.

### `persistColumnOrder` (store.ts:192-215)
Uses `Promise.all(writes)` where each write is a task file update. If any write fails mid-operation, some tasks have new order and others have old. No transactional behavior — partial writes are allowed.

## Why This Needs Fixing

**Severity: High**

State inconsistency leads to confusing UX — the board shows one thing, but caches show another. On page reload, the actual file state is loaded, which could differ from what the user was seeing. This creates a situation where the user thinks their action failed (due to toast) but the UI shows partial success.

## How to Fix

### Step 1: Capture All State for Rollback

Every mutating action should capture the complete previous state before any changes:

```typescript
createTask: async (title, columnName) => {
  const { columns, taskContents, searchMatches, tasksDirHandle, config } = get();
  
  // Capture ALL state that could be mutated
  const previousColumns = columns;
  const previousTaskContents = taskContents;
  const previousSearchMatches = searchMatches;
  const previousBoardTitle = boardTitle;
  
  // ... generate ID, create task ...
  
  // Optimistic updates
  const newColumns = columnName === 'backlog' ? [...columns] : /* ... */;
  const newContents = new Map(taskContents);
  newContents.set(id, { frontmatter: fm, body });
  
  set({ 
    columns: newColumns, 
    taskContents: newContents,
    // searchMatches not updated on create (no search content yet)
  });
  
  try {
    await fsWriteTaskFile(tasksDirHandle, id, fm, body);
    get().toast(`Created ${id.replace(/^t/, '')}`);
    return id;
  } catch (e) {
    get().toast('Failed to create: ' + (e as Error).message, 'error');
    // Rollback ALL mutated state
    set({ 
      columns: previousColumns,
      taskContents: previousTaskContents,
      searchMatches: previousSearchMatches,
    });
    return null;
  }
}
```

### Step 2: Transaction Pattern for persistColumnOrder

Replace `Promise.all` with sequential writes and rollback on failure:

```typescript
async function persistColumnOrder(
  tasksDirHandle: FileSystemDirectoryHandle,
  affected: Array<{ id: string; status: string; order: number }>,
  columns: BoardColumn[]
): Promise<void> {
  // Capture original state for rollback
  const originalStates = new Map<string, { status: string; order: number }>();
  
  for (const task of affected) {
    originalStates.set(task.id, { status: task.status, order: task.order });
  }

  try {
    for (const task of affected) {
      // Read current file
      const taskResult = await readTaskFile(tasksDirHandle, task.id);
      if (!taskResult.ok) throw new Error(`Cannot read task ${task.id}`);
      
      // Update frontmatter
      const updated = {
        ...taskResult.value.frontmatter,
        status: task.status,
        order: task.order,
      };
      
      // Write back
      await fsWriteTaskFile(tasksDirHandle, task.id, updated, taskResult.value.body);
    }
  } catch (e) {
    // Rollback: restore original status/order for all affected tasks
    for (const [id, original] of originalStates) {
      try {
        const taskResult = await readTaskFile(tasksDirHandle, id);
        if (taskResult.ok) {
          const restored = { ...taskResult.value.frontmatter, status: original.status, order: original.order };
          await fsWriteTaskFile(tasksDirHandle, id, restored, taskResult.value.body);
        }
      } catch {
        // Rollback of rollback — log but don't throw
        console.error(`Failed to rollback task ${id}`, e);
      }
    }
    throw e; // Re-throw original error
  }
}
```

### Step 3: Add "Pending Operations" State

Track ongoing mutations so UI can show loading state and prevent conflicting operations:

```typescript
interface StoreState {
  // ...
  pendingOperations: Set<string>;  // e.g., 'move:t1', 'create:t5'
}

// Before mutation:
set({ pendingOperations: new Set([...pendingOperations, `move:${taskId}`]) });

// After completion (success or failure):
const newPending = new Set(pendingOperations);
newPending.delete(`move:${taskId}`);
set({ pendingOperations: newPending });
```

This prevents race conditions where two operations conflict.
